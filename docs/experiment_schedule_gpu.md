# SSRS 统一跨域多模态分割项目：实验排期表（论文交付版 v1.1）

## 1. 目标与总原则
- 目标：基于 MFNet 完成一条可投稿的弱跨域多模态遥感分割主线，而不是继续无序叠加模块。
- 主线数据：ISPRS Vaihingen 和 Potsdam。
- 主结论来源：weak-cross-domain 结果，不是 same-domain 单次最优值。
- same-domain 作用：机制验证，特别是验证边界和对象结构约束是否真的有作用。
- 执行转向（自第 6 周起）：先做 same-domain 机制闭环与稳态统计，再回到 weak-cross-domain 做迁移复验。
- LoveDA 定位：附加压力测试或附录材料，不作为第一轮验收门槛。
- GPU 预算口径：1 GPU-day = 1 张 GPU 连续训练 24 小时。

## 2. 当前版本的实验组织原则
结合当前正式版和 v1.1 版本，实验排期采用“主线必做 + 条件增强”结构：

### 2.1 主线必做
- M0：MFNet + single-level high-level UDA。
- M0+S：在 M0 基础上加入边界与对象结构约束。

### 2.2 条件增强
- E1：multi-level UDA，即 mid+high 对齐，只在 single-level 已稳定时进入。
- E2：prompt-guided 伪标签精炼，只在 UDA 或结构主线已稳定时进入。

这样安排的目的是防止第 3 周到第 5 周同时引入过多变量，导致最后无法解释到底是哪一项有效。

## 3. 最终论文需要的结果清单

### 3.1 主表
- T1：same-domain 定量对比表。
- T2：weak-cross-domain 主结果表。

### 3.2 分析表
- T3：组件消融表，至少包含 -adv、-bdy、-obj、-DSM。
- T4：模态与微调策略分析表，至少包含 RGB、RGB+DSM、LoRA、Adapter。
- T5：效率分析表，至少包含参数量、可训练参数、显存、吞吐或推理时间。
- T6：稳定性或数据效率分析表，至少包含不同数据比例或不同随机种子统计。
- T7：条件增强分析表，优先用于 single-level 与 multi-level 对齐；若 prompt 分支保留，则增加 prompt 消融子表。

### 3.3 图像类结果
- F1：方法框架图。
- F2：same-domain 与 weak-cross-domain 可视化对比图。
- F3：道路、建筑等边界放大图。
- F4：训练与验证收敛曲线图。
- F5：伪标签质量图、热图或特征分布图。
- F6：failure cases 图。

## 4. 设备假设
- 训练卡：1 到 2 张 24GB 显存级别 GPU。
- 周可用时长：7x24 小时。
- 若只有 1 张卡，整体周期建议增加 20% 到 35%。
- 若 weak-cross-domain 三种子无法全部完成，则优先保证主线单种子 + 小规模稳定性验证，再补第三个种子。

## 6. 每周详细执行清单与交付物

### 第 1 周：基线冻结周（2.0 GPU-day）
目标：建立后续所有实验的统一起点。

工作项：
- 固定数据协议、窗口大小、stride、batch size、学习率和随机种子策略。
- 跑通 MFNet same-domain 基线，优先保证 Vaihingen 和 Potsdam 各至少 1 次完整结果。
- 固定 csv、train.log、summary markdown 和图表输出格式。

本周必须交付：
- T1-v1：基线复现表。
- F4-v1：MFNet 训练损失与验证指标曲线。
- 附加结果：2 组代表性预测图，作为可视化模板。

本周验收门槛：
- MFNet 复现误差控制在历史值或论文值的约 1.0 mIoU 以内。
- 输出目录、字段和命名规范固定。

### 第 2 周：same-domain 结构机制验证（2.5 GPU-day）
目标：验证结构约束是否真的改善边界与对象完整性，而不是只看总分。

工作项：
- 在 MFNet same-domain 设置下分别测试 Baseline、+Boundary、+Object、+Boundary+Object。
- 记录道路、建筑等边界敏感类别的单类结果。
- 采样 2 到 3 个典型 patch 做放大图对比。
- 若时间允许，做 prompt 分支的局部可用性检查，但不纳入正式主线判断。

本周必须交付：
- T3-v1：same-domain 组件消融表。
- F3-v1：边界放大图。
- F4-v2：seg、bdy、obj 损失分解曲线。

本周验收门槛：
- 即使 overall mIoU 不明显提升，也至少要证明边界类指标或局部结构更好。
- 若结构项所有指标都负向，则其角色从“主线候选”降级为“条件分析分支”。

### 第 3 周：弱跨域最小可行版（3.0 GPU-day）
目标：建立 weak-cross-domain 的最小正增益原型。

工作项：
- 选择一个主协议，例如 Vaihingen 到 Potsdam，先固定方向。
- 跑 source-only 基线。
- 在 MFNet 高层特征处加入 single-level feature-level 对抗对齐。
- 若余量允许，只做 prompt feasibility check，不进入主结果表。

本周必须交付：
- T2-v1：weak-cross-domain 主结果初稿。
- F2-v1：target 域可视化对比图。
- F6-v0：收集至少 2 类失败案例备用。

本周验收门槛：
- +UDA 至少在 target 域相对 source-only 方向上为正。
- 若 +UDA 仍明显劣于 source-only，则先不叠结构约束，也不做 prompt 主线实验。

### 第 4 周：UDA 稳定化与层级筛选（3.5 GPU-day）
目标：把第 3 周的原型变成可复验结果，同时判断是否需要多层对齐。

工作项：
- 调整 $\lambda_{adv}$、GRL 强度、判别器学习率、warmup 轮数。
- 对当前最优 single-level 配置做 2 到 3 个随机种子复跑。
- 对比 single-level 与 multi-level 对齐，判断中层对齐是否真的值得保留。

本周必须交付：
- T6-v1：稳定性与超参数表。
- T7-v1：single-level vs multi-level 对齐表。
- F4-v3：收敛稳定性图。
- T2-v2：带 mean±std 的 UDA 主结果表。

本周验收门槛：
- 至少有 1 组 UDA 配置在 2 到 3 个种子下方向一致。
- 若 multi-level 无明显收益或明显不稳，则正式主线保留 single-level。

### 第 5 周：联合框架验证周（3.0 GPU-day）
目标：在已稳定的 UDA 主线上判断结构约束是否保留，以及 prompt 分支是否值得进入正文。

工作项：
- 基于第 4 周最优 UDA 配置，加入 boundary/object 结构约束。
- 评估不同伪标签阈值或高置信掩码策略。
- 若 M0 或 M0+S 已稳定，再引入 prompt-guided 伪标签精炼，并与固定阈值伪标签直接对比。

本周必须交付：
- T2-v3：Source-only、+UDA、+UDA+BDY、+UDA+OBJ、+UDA+BDY+OBJ 主结果表。
- F3-v2：target 域边界放大图。
- F5-v1：伪标签质量图、边界响应图或 feature heatmap。

本周验收门槛：
- 若结构约束在总指标或边界类指标上有稳定正增益，则保留为主模型组成部分。
- 若 prompt 分支无稳定正增益，则暂不进入主贡献，只保留中间记录。

### 第 6 周：转向同域闭环周（4.0 GPU-day）
目标：先把机制在 same-domain 场景里验证清楚，再决定哪些模块值得回灌到跨域主线。

工作项：
- same-domain 多种子复验（建议至少 3 seed），输出 mean±std。
- 组件消融：-bdy、-obj、-DSM（同域优先）；-adv 可作为附加项记录。
- 模态分析：RGB-only 与 RGB+DSM（优先补齐）。
- 微调策略分析：冻结、LoRA、Adapter（至少形成初版可比表）。
- 针对高/低植被混淆，补类别级混淆矩阵和边界局部对比图。

本周必须交付：
- T3-final：组件消融表。
- T4-v1：模态与微调策略分析表（至少初版）。
- F3-v3：同域边界放大与困难类别对比图。
- F4-v4：同域多 seed 收敛稳定性图。

本周验收门槛：
- 每个拟保留模块都要在同域形成“可解释证据”，不是只看总体 mIoU。
- 若某模块在同域无法形成稳定收益或现象优势，不进入第 7 周跨域回灌名单。

### 第 7 周：跨域回灌验证周（3.0 GPU-day）
目标：把第 6 周验证通过的机制回灌到 weak-cross-domain，判断是否具备稳定迁移收益。

工作项：
- 固定一组 UDA 基线 + 一组机制增强配置，做 2 到 3 个 seed 复验。
- 对比 single-level 与 multi-level，仅在第 6 周证据支持时启用 multi-level。
- 若 prompt 分支在第 6 周未过门槛，本周不引入 prompt 主线对比。
- 输出跨域稳定性统计（mean±std）与失败案例更新。

本周必须交付：
- T2-stable：跨域主结果稳定版（mean±std）。
- T6-v2：稳定性更新表（同域与跨域并列）。
- T7-v2：single-level 与 multi-level（或 prompt）条件增强对比更新。
- F5-v2：伪标签质量/边界响应更新图。

本周验收门槛：
- 至少 1 组增强配置在跨域中保持方向一致的正向收益。
- 若增强项跨域不稳定，则降级为分析分支，不进入第 8 周主模型。

### 第 8 周：封板复验周（1.5 GPU-day）
目标：整理出可直接写论文的完整资产包。

工作项：
- 对最终主模型做 1 次完整复训。
- 统一所有表格字段、图题和实验设置描述。
- 明确 prompt 分支最终去留。
- 汇总同域、弱跨域、消融、效率、可视化和失败案例。

本周必须交付：
- T1 到 T7 定稿。
- F1 到 F6 定稿。
- 一份结果索引文档，标明每张表和每张图来自哪个 run 目录。

本周验收门槛：
- 主表、分析表、效率表和图像证据能够彼此闭环。
- 能够写出“方法为什么有效”，而不是只写“结果更高”。

## 7. 实验过程中必须保存的数据
- 实验元数据：实验编号、数据协议、模型版本、git commit、seed、时间戳。
- 配置快照：训练配置 yaml、命令行参数、环境版本信息。
- 训练过程：iter 和 epoch 级别的 loss 与 metric csv。
- 伪标签统计：置信度分布、有效像素占比、类别占比变化。
- 域对齐统计：域判别器准确率、各层对抗损失。
- prompt 统计：若启用 prompt，保存提示数量、类型和有效区域比例。
- 模型与状态：best/last 权重、优化器、学习率调度器状态。
- 推理输出：预测图、误差图、边界图、按类指标明细。

## 8. 阶段性验收标准

### 阶段 A：第 1 到第 2 周
- MFNet 基线稳定复现。
- 结构损失至少在某类机制层面有证据，不要求此时就是最终主模型。

### 阶段 B：第 3 到第 4 周
- weak-cross-domain 的 UDA 原型成立。
- 存在至少一组稳定配置在 target 域带来正增益。
- 已明确 single-level 是否优于或不逊于 multi-level。

### 阶段 C：第 5 到第 6 周
- 明确同域机制闭环：哪些组件在 same-domain 具备稳定证据。
- 形成第 7 周跨域回灌白名单（仅保留通过门槛的组件）。
- prompt 分支在本阶段仅做是否保留的门槛判断。

### 阶段 D：第 7 到第 8 周
- 跨域回灌结论稳定（含 mean±std），并形成最终主模型路线。
- 效率与数据效率分析齐备。
- 所有主表主图可直接进入论文正文或附录。

## 9. 风险缓冲与转向规则
- 预留 3.0 GPU-day 作为风险缓冲。

主要风险：
- 对抗训练振荡。
- 伪标签噪声导致结构损失反向破坏。
- multi-level 对齐收益不稳定。
- prompt 分支增加复杂度但没有形成稳定增益。
- 多随机种子方差过大。

转向规则：
- 若第 4 周结束时 UDA 仍不稳定，主线应简化为“多模态参数高效微调 + 结构增强”的 same-domain 与弱迁移分析论文，而不是强行保留对抗分支。
- 若 multi-level 对齐不稳定，则正式版本只保留 single-level 对齐。
- 若第 5 周 prompt 分支没有稳定收益，则降级到附录或删除，不拖累主线。
- 若第 6 周同域证据不足，则第 7 周暂停跨域新增实验，优先补同域机制证据。
- 若第 7 周效率表现较差，则补强“数据效率”或“边界质量”作为第二卖点。

## 10. 可选附加阶段：LoveDA 压力测试
- 建议在主线完成后追加 1 到 2 周。
- 目标：展示方法在更强域偏移场景下的鲁棒性，而不是追求绝对最高分。
- 推荐产出：附录表格 + 1 张代表性可视化图 + 1 组 failure case。
- 呈现方式：强调相对提升、边界质量和稳定性，不把 LoveDA 结果作为主结论唯一依据。

## 11. 建议周内工作节奏
- 周一到周二：启动主训练与对照实验。
- 周三：检查中期结果，判断是否继续或提前止损。
- 周四到周五：补跑消融、保存图表和整理失败案例。
- 周末：整理本周 T/F 编号结果，更新下周优先级。

建议执行要求：
- 每周结束前必须至少产出 1 张表和 1 张图。
- 每项实验都要写一句“它打算证明什么”。
- prompt 和 multi-level 分支不能在未过门槛前默认进入主贡献。
